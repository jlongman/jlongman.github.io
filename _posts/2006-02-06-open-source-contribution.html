open source contribution

I was playing with PCRCommander, a software XM tuner/recorder written in Python, a language I'm interested in. I realized that it had a few memory leak problems and a few weird UI issues (you can't resize the window!) that were irritating. Well the beauty of open-source is scratching itches. (... the beauty of open sores is scratching itches too)

Fine, well, good, I'll get the python plugin, PyDev, for Eclipse, the next kitchen-sink-wannabe. No problem. Now check out the source code... Oh, it uses darcs, a decentralized version control system that is based around patches. Which I have never used.

Well, I wonder if Eclipse has a plugin for Darcs? Yes, it does, and it's called eclipsedarcs. Fine, oh, I need to install a haskell compiler in order to get darcs to run. Fine, if there's nothing else I learned at Ingenia, it's how to compile and run stuff without knowing what it is or does. Oh, except there's an OSX pre-compiled distribution. Wonderful. Install everything.

Hmm... I was expecting eclipsedarcs to be a little more bouncy when I entered the URL for the repository. Researchresearch. Ah, it can't find darcs on the path, and you have to play with OSX packaging properties in order to get it to work. Sounds too 'simple'. Actually, it sounds like a PITA! Why can't I just specify the executable!?

Get the eclipsedarcs plugin source code using the darcs command line. Import it into eclipse. Find the code which executes the darcs executable. Ah, they've already added a preference for the executable, but they just haven't exposed it in the UI, and also, it's exactly like the preference for the user's email - a simple text line. Sounds like a job for scratchy people!

Make the changes, try to figure out how multiple plugins are tested in Eclipse. Try to get a Feature to run. No. Try to create an Update site. Yes. Try to build the Update site. No, 'can't find the plugins'. Generally you run a new instance of Eclipse with the plugin you've just added in order to test it, but the 'hello world'/'special paste' one I wrote {am still writing} was a single plugin, so I didn't know if you need a feature to get multiple plugins going in the test Eclipse environment.

Screw it, run the plugin and trust the dependencies to be loaded. Ah! It worked, except for the text being screwed up. Hopefully just a packaging problem.

Cool, well, everything doesn't quite seem to be here. I can get change histories from the repositories, but can't seem to 'get' the PCR Commander repository. Maybe there's something else wrong, or maybe that's all the plugin does so far (it is, after all, only version 0.2.0).

Hoopy, I wonder if anyone else wants my little mod? A little searching shows the exact link that showed me something about OSX packaging properties and it's previous is an eclipsedarcs developer saying they're too busy to do what I think I just did.

Excellent.

Okay, so eclipsedarcs isn't apparently showing me how to feed things back to the source. So back to the command line. 'whatsnew': handy, but I don't want to check in my maulings of the plugin configuration files, nor the .class files... 'revert', great! 'record', wunderbar! Okay. Now how do I send it, considering I don't have an exposed web repository? Email patch! Sounds easy.

Oh, smart, you need GPG, a cryptographic signing library, to authenticate the patch. Fortunately someone else in OSX land has packaged it up. ‘kay, build the key, install it into the right directory. Figure out who to send it to... and bang. Done.

All those steps and I still haven't even checked out PCRCommander, and it's now time to walk the dogs and sleep. I'm definitely standing on the shoulders of giants, here, but its a good thing they were there or we'd ever get nothing done!
